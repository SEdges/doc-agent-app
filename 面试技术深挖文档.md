## 📋 项目一句话介绍

**场景**：Joern 是一款代码分析工具，但官方文档分散在数十个网页中，查找信息效率低下。

**方案**：构建了一个 RAG（检索增强生成）系统，递归爬取全部文档 → 向量化存储 → 支持多轮语义问答。

**成果**：将平均 5-10 分钟的「打开文档 → Ctrl+F → 跳转」降低到 3-5 秒的自然语言问答。

### 问题 1：为什么选择 BFS 递归爬取而不是用 Sitemap？

**问题背景**：
- docs.joern.io 没有提供 sitemap.xml
- 首页只有部分导航链接，不包含所有文档页面

**我的技术决策**：
```
采用广度优先搜索（BFS）递归抓取：
1. 从首页开始，提取所有同域名链接
2. 维护 visited 集合去重
3. 继续访问新链接直到没有新页面
4. 设置 max_pages=100 防止无限循环
```

**深挖点 - 为什么不用 DFS？**
```
BFS 优势：
- 优先抓取「距离首页近」的核心文档
- 即使中断也能保证最重要的内容已被抓取
- 更符合网站信息架构（核心内容链接深度浅）

DFS 风险：
- 可能先深入边缘内容
- 某个分支出错会影响整体进度
```

**遇到的坑**：
```python
# 第一版代码（有 Bug）- 只抓到 5 个页面

    if href.startswith(base_url):
        links.add(href)  # 没有递归！

# 修复后 - 抓到 30-100 个页面
while to_visit and count < max_pages:
    current_url = to_visit.pop()
    # 递归提取新链接...
```

- 如何处理 JavaScript 渲染的动态内容？

### 问题 2：chunk_size=1200 和 chunk_overlap=100 这两个参数怎么定的？

**问题背景**：
- 文档分块是 RAG 系统的关键，直接影响检索质量
- 太大：检索到无关内容；太小：丢失上下文

**我的技术决策**：
```
chunk_size=1200 的理由：
1. 中文 + 代码混合文档，平均段落 600-800 字符
2. 1200 能容纳 1-2 个完整段落 + 代码块
3. 经验值：控制在 LLM context window 的 1/4 以内

chunk_overlap=100 的理由：
1. 8% 重叠率，保证跨块内容不丢失
2. 对于「问题 A 的答案在第 1 块末尾 + 第 2 块开头」的情况有容错
```

**实验验证**：
```
测试问题："CPG 的节点类型有哪些？"

chunk_size=500:  检索到 3 个块，但每块只有部分列表
chunk_size=1200: 检索到 1 个完整块，包含全部节点类型
chunk_size=2000: 检索到 1 个块，但包含大量无关内容
```

**可追问**：
- 如果是纯代码仓库（而非文档），分块策略会怎么变？
- 如何动态调整 chunk_size？能否做到「按语义边界」分块？
- overlap 设太大会有什么问题？

### 问题 3：为什么检索 k=3 而不是 k=5 或 k=10？

**问题背景**：
- k 太小：可能漏掉相关信息
- k 太大：引入噪声，且增加 LLM 输入 token 成本

**我的技术决策**：
```python
retriever = db.as_retriever(search_kwargs={"k": 3})
```

**定 k=3 的理由**：
```
1. 成本约束：
   - gpt-4o-mini 的 input token 价格 = $0.15/1M tokens
   - 每个 chunk 约 1200 字符 ≈ 400 tokens
   - k=3 时每次检索消耗 1200 tokens
   - k=10 时消耗 4000 tokens（成本 3x）

2. 质量实测：
   - Joern 文档主题明确，相似度 Top 1 通常就是答案
   - k=3 作为冗余足够
   - k=5 以上新增的块相关性显著下降

3. 响应速度：
   - k 越大，LLM 处理时间越长
   - k=3 时响应时间 2-3s，k=10 时 5-7s
```

**可追问**：
- 如果用户问的是「跨多个文档」的综合性问题，k=3 够吗？
- 能否动态调整 k？比如先检索 k=1，如果置信度低再扩大？
- 除了 Top-k，还有什么检索策略？（MMR、ReRank）

### 问题 4：多轮对话的上下文是怎么实现的？

**问题背景**：
- 用户可能说「它支持哪些语言？」，这里的「它」指前一轮提到的 Joern
- 需要让 LLM 理解对话上下文

**我的技术决策**：
```python
# 方案：将历史对话作为 prompt 的一部分传入
def answer_with_context(query, chat_history=None):
    history_text = format_chat_history(chat_history)
    
    prompt = ChatPromptTemplate.from_messages([
        ("system", 
         "=== 对话历史 ===\n{history}\n\n"
         "=== 相关文档 ===\n{context}"),
        ("user", "{question}")
    ])
```

**为什么不用 LangChain 的 ConversationChain？**
```
1. ConversationChain 会把全部历史塞进 prompt
   - 10 轮对话后 token 成本爆炸
   
2. 我的方案：只传最近 N 轮 + 当前检索结果
   - 历史用于理解指代
   - 检索结果用于回答问题
   - 职责分离，可控性强
```

**遇到的坑**：
```
场景：用户连续问 5 个关于 CPG 的问题
问题：第 6 个问题检索结果和历史完全无关

原因：历史太长导致 LLM 「分心」，反而忽略当前检索结果
修复：限制历史长度，只保留最近 5 轮
```

**可追问**：
- 怎么判断用户「换话题了」？需要清空历史吗？
- 如果历史中有错误答案，会不会污染后续回答？
- 能否让 LLM 自己决定哪些历史是相关的？

### 问题 5：为什么用 WebBaseLoader 而不是 Playwright？

**问题背景**：
- Playwright 可以抓取 JavaScript 渲染的动态内容
- WebBaseLoader 只能抓取静态 HTML

**我的技术决策**：选择 WebBaseLoader

**原因（按优先级）**：

```
1. 兼容性坑（决定性因素）：
   - Windows + Streamlit + Playwright 有严重的事件循环冲突
   - 错误: "RuntimeError: This event loop is already running"
   - 在 Streamlit 的异步环境中无法正常启动 Playwright

2. 目标网站特性：
   - docs.joern.io 是静态站点生成（SSG）
   - 所有内容在 HTML 中，无需 JS 渲染

3. 资源消耗：
   - Playwright 需要启动真实浏览器进程
   - 100 个页面 = 100 次浏览器启动
   - WebBaseLoader 只是 HTTP 请求，快 10 倍以上
```

**如果目标是动态网站怎么办？**
```
方案 A：把爬取和问答分离
- 单独的爬虫脚本使用 Playwright
- 爬完后导入向量数据库
- Streamlit 只负责问答

方案 B：使用无头浏览器服务
- 部署 Browserless 或 Playwright Server
- 通过 API 调用而非本地进程
```

**可追问**：
- 你怎么发现 Playwright 和 Streamlit 冲突的？排查过程是什么？
- 如果文档更新频繁，如何设计增量爬取？
- 怎么处理反爬机制（验证码、频率限制）？

### 问题 6：Chroma 的选型理由是什么？为什么不用 FAISS 或 Pinecone？

**问题背景**：
- 向量数据库选型直接影响性能、运维复杂度、成本

**我的决策矩阵**：

| 维度 | Chroma | FAISS | Pinecone |
|------|--------|-------|----------|
| 部署复杂度 | 零配置（SQLite） | 需手动管理索引 | 需云服务账号 |
| 本地运行 | ✅ 原生支持 | ✅ 支持 | ❌ 仅云端 |
| 持久化 | 自动（0.4+） | 需手动 save/load | 自动 |
| 数据量级 | 万级最佳 | 亿级也行 | 无限扩展 |
| 成本 | $0 | $0 | 按用量付费 |

**选 Chroma 的理由**：
```
1. 项目规模匹配：
   - 100 个页面 × 10 chunks = 1000 个向量
   - Chroma 对万级数据绰绰有余

2. 开发体验：
   - pip install langchain-chroma 一行搞定
   - 无需配置 Docker/K8s

3. LangChain 集成最好：
   - 原生 from_documents() 方法
   - 其他库需要额外封装
```

**遇到的坑**：
```python
# Chroma 0.3 需要手动调用 persist()
db.persist()  # 旧版必须

# Chroma 0.4+ 自动持久化，调用 persist() 会 warning
# 我的代码一度写了 persist()，升级后才发现多余
```

**可追问**：
- 如果数据量增长到百万级，Chroma 还能用吗？
- 向量检索的时间复杂度是多少？Chroma 用的什么索引结构？
- 如果要支持「删除某个文档」，Chroma 能做到吗？

### 问题 7：系统的错误处理策略是怎么设计的？

**问题背景**：
- 网络请求可能超时
- 某个页面可能抓取失败
- API Key 可能失效

**我的设计原则：局部失败不影响整体**

**爬虫层错误处理**：
```python
try:
    resp = requests.get(current_url, timeout=10)
    resp.raise_for_status()
except Exception as e:
    print(f"Error crawling {current_url}: {e}")
    continue  # 跳过这个页面，继续下一个
```

**设计理由**：
```
1. 文档爬取是「尽力而为」场景
   - 抓到 95 个页面和 100 个页面差别不大
   - 不应该因为 1 个页面失败导致全部重来

2. 用户可见的错误只在关键路径抛出：
   - API Key 无效 → 阻塞启动（必须解决）
   - 单个页面失败 → 静默跳过（可接受损失）
```

**可追问**：
- 如果 OpenAI API 调用偶发失败（网络抖动），你怎么处理？重试策略是什么？
- 怎么让用户知道哪些页面抓取失败了？
- 如果向量数据库损坏了，有备份/恢复机制吗？

### 问题 8：性能瓶颈在哪里？做过哪些优化？

**问题背景**：
- 用户期望秒级响应
- 但系统涉及网络 IO（OpenAI API）和计算（向量检索）

**性能分析**：
```
用户提问 → 答案显示 的时间分解：

1. 向量检索：~50ms（Chroma 本地查询，快）
2. OpenAI Embedding：~200ms（把 query 向量化，需要调 API）
3. OpenAI Chat：~2000ms（生成答案，主要瓶颈）
4. Streamlit 渲染：~10ms

总计：~2.3s，其中 85% 时间花在 OpenAI API
```

**优化手段**：

```python

@st.cache_resource  # 向量数据库只构建一次
def build_vector_db(documents, api_key):
    ...

@st.cache_data  # 爬取结果缓存
def crawl_joern_docs(base_url):
    ...

# 2. 选择 gpt-4o-mini 而非 gpt-4
# gpt-4o-mini 响应速度快 3 倍，成本低 10 倍

# 3. 检索 k=3 而非 k=10
# 减少 LLM 输入 token，加速生成
```

### 问题 9：如果让你重新设计，会有什么不同？

**反思 1：爬取和问答应该解耦**
```
现状：Streamlit 应用里包含爬虫逻辑
问题：
- 爬取 100 个页面需要 3-5 分钟
- 用户每次重建都要等
- 难以做增量更新

改进方案：
- 爬取做成独立 CLI 工具
- 定时任务自动更新
- Streamlit 只负责问答
```

**反思 2：应该加入 ReRank**
```
现状：只用向量相似度 Top-k
问题：
- 向量相似度不等于「语义相关度」
- 可能检索到表面相似但内容无关的块

改进方案：
- 引入 Cross-Encoder ReRank
- 先粗检索 k=10
- 再用 ReRank 模型精排到 k=3
```

**反思 3：应该支持文档溯源**
```
现状：只返回答案，不告诉用户「来自哪个页面」
问题：
- 用户无法验证答案准确性
- 无法深入阅读原文

改进方案：
- 保存 chunk 的 source URL 元数据
- 在答案后附上「参考来源」链接
```

**可追问**：
- ReRank 模型和 Embedding 模型有什么区别？
- 如何评估 RAG 系统的质量？有什么指标？
- 如果文档之间有矛盾（旧版和新版），怎么处理？

### 问题 10：这个项目最有价值的工程经验是什么？

**经验 1：先验证假设再写代码**
```
踩坑：以为「首页包含所有文档链接」
结果：第一版只抓到 5 个页面
教训：应该先手动数一下文档页面数量

正确做法：
1. 先手动探索目标网站结构
2. 验证假设（有几个页面？链接在哪？）
3. 再动手写代码
```

**经验 2：选型要考虑「组合兼容性」而非「单点最优」**
```
踩坑：Playwright 是最强的爬虫工具，但和 Streamlit 冲突
教训：技术选型不是选「最好的」，而是选「能一起工作的」

正确做法：
1. 先确定技术栈的「固定约束」（必须用 Streamlit）
2. 在约束内选择「兼容且够用」的方案
```

**经验 3：错误处理要区分「致命」和「可容忍」**
```
设计：
- API Key 无效 → 致命，阻塞启动
- 单页面抓取失败 → 可容忍，静默跳过
- 向量数据库文件损坏 → 可恢复，提示重建

原则：用户能解决的问题要清晰报错，用户解决不了的问题要自动容错
```
